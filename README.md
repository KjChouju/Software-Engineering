[1]个人本地仓库提交规范
(1)每次提交都应该是逻辑上独立，可理解，且完整的变更单元
原子性提交：一次提交只做一件事。例如，修复一个Bug 或 添加一个新功能 或 重构一个函数。避免将多个不相关的修改（如一个新功能和一个Bug修复）混在一次提交中。
提交信息规范化：使用约定式提交
（类型）：增加微信扫码登录功能
修改登陆页面UI，增加扫码入口
(2)功能开发，Bug修复应与主分支隔离，在独立的分支上进行
(3)在推送到远程前，整理本地历史，使其清晰线性
在将本地功能分支合并到主分支（如 main）之前，先执行 git rebase main
(4)频繁的从上游拉去更新，避免大的集成冲突，每两周上传一次代码
git fetch + git rebase origin/main：推荐方式。获取远程更新并将你的工作变基到其上
[2]项目组远程仓库管理原则与规范
(1)核心分支必须受到严格保护，除了仓库管理员，任何人不能直接向保护分支推送代码，所有更改必须通过拉取请求合并
(2)PR是代码进入主库的唯一大门，是质量保证的关键环节，填写清晰的PR模板，包括：修改目的、修改内容、测试情况、相关Issue链接
(3)分级权限：
管理员：核心维护者，负责仓库设置、分支保护、处理关键Bug
写入者：普通开发者，可以创建分支、推送代码、创建PR
只读者：外部贡献者或实习生
(4)所有正式发布版本都应在main分支上打上标签，标签名遵循语义化版本控制（如v1.2.3）
